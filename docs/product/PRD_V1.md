# 第一部分：总览

## 一、产品定位与价值主张

### 1.1 一句话定位
> 让个人和小团队用「招人 + 派活」的方式管理 AI，把零散的 Agent 调用沉淀成一支可复用的数字团队。

### 1.2 与同类产品的差异
| 维度 | 主流 AI 助手 / Agent | AI 数字员工 |
|---|---|---|
| 心智模型 | 「对话」/「指令」 | 「组织 + 岗位」 |
| 复用单元 | Prompt / Agent 配置 | **员工档案**（岗位+技能+记忆+履历） |
| 工作流入口 | 用户主动提问 | **以工单管理需求**（澄清前置 → 执行 → 验收） |
| 工程规范 | 通常无规范约束 | **项目规范 conventions + AGENTS.md 兼容**，硬约束驱动执行 |
| 多任务管理 | 散落在不同会话中 | 多员工各自档案 + 工单列表统一调度 |
| 沉淀资产 | 散落的会话记录 | 员工成长档案、项目知识库、需求闭环 |

> 注：V1.0 为手动指派、同步执行；**自动分派、异步并行**为 V1.1+ 能力。

### 1.3 目标用户与场景
- **独立开发者**：组建「产品经理 + 前端工程师 + 设计师」三人 AI 团队，并行推进副业项目
- **创业者 / 一人公司**：用「运营专员 + 客服 + 数据分析师」处理重复性工作
- **自由职业者**：把不同领域客户的活儿分给对应「专员」，按需求工单管理交付

### 1.4 核心形态
- **真·AI 员工执行平台**：员工能调用 LLM 和工具，真实完成任务
- **核心数据关系**：员工 → 技能 → 接需求；**项目承载知识**（项目介绍自动 RAG）
- **计费形态**：V1.0 用户自带 LLM Key
- **命名与国际化备注**：中文语境用「员工 / Employee」，海外社区主流叫「Agent / Worker / Assistant」。出海版本对外发声时优先用 "AI Worker" 或 "AI Agent"，避开 "AI Employee" 在部分地区"取代人类劳动"的负面联想。内部文档与数据模型字段保持英文 `Employee`，对外营销文案按地区调整。

## 二、产品哲学

> **宁可多问，不可盲干。**
> 员工不是"接到指令就闷头跑"的工具人，而是会沟通、会确认、会暴露思考过程的协作者。
> 每一步对用户可见、可介入；犯过的错会成为下次的提醒。

体现在三个机制：
1. **澄清前置**：接单先复述理解 + 列拆解 + 提问，确认后才动手
2. **过程透明**：思维链 + 当前步骤 + 下一步计划三栏实时可见
3. **纠错沉淀**：失败 / 返工 / 差评自动复盘，写入个人教训 + 项目踩坑库

## 三、V1.0 — 引擎验证版（两阶段周期）

**定位**：纯引擎，验证核心闭环；**单用户本地使用**；用户自带 LLM Key；**不做商业化**。

| 阶段 | 周期 | 目标 | 退出标准 |
|---|---|---|---|
| **V1.0-α Demo** | 4 周 | 端到端跑通"接需求 → 澄清 → 执行 → 沉淀"主路径，单需求场景稳定 | §12 验收主路径 #1–#6 通过 |
| **V1.0-β Stable** | +4~8 周 | 多需求并发、持久化、Budget Cap、Context 处理、规范、Governance 全部稳定 | §12 全部 17 条验收通过 + 量化指标达成 |

> **周期重估说明**：原"4 周做完所有 P0" 严重低估——仅"状态实时落库 + 思维链流式 UI + 多需求并发"前端工程量就接近 4 周（2 人）。两阶段策略避免"4 周 demo 当稳定版交付"的陷阱。

**α 阶段包含**（14 项 P0-α，详见 §13）：项目/员工/技能/需求 CRUD、澄清前置 + 思维链三栏、`ask_user` 工具调用、执行中再澄清（2 种触发）、状态实时落库、Budget Cap、工具错误处理（超时重试）、项目介绍自动 RAG、项目规范 conventions（required）、双向沉淀 + 自动注入、工具注册地基（仅 `ask_user`）、Token 成本可见、Thread/Message 模型、LLM Key OS 凭证加密。

**β 阶段追加**（13 项 P0-β + 4 项 P1）：多需求真并发、复杂度阈值、再澄清扩到 5 种、Context 摘要、工具错误处理全套、Prompt Cache、recommended 规范 RAG、AGENTS.md 文件读取、Importance Scoring、分类置信度暂存、Onboarding 样板项目、Governance 完整、自带工具扩展、Replay & Debug 等。

**全程不做**（V1.0 整体）：仪表盘、计费、用户系统、模板市场、异步执行、日报周报、员工经理、自动推荐、Trigger 事件驱动、Git/PR/CI 集成、完整 codebase 索引、跨项目共享 Brain。完整清单见 §11。

## 四、关键决策记录（V1.0 范围）

> 只记录"有 trade-off、选了 A 而非 B"的决策；"V1.0 不做某事"的清单见 §11。

| # | 议题 | 决策 | 备选与放弃理由 |
|---|---|---|---|
| 1 | 记忆主体 | **以项目为主**：业务知识沉淀归项目，员工只保留自我画像 + 个人教训 | 备选「以员工为主」放弃：员工跨项目会带走业务知识，违反保密性 |
| 2 | 项目介绍 RAG | 项目介绍/知识**自动向量化**，员工接需求时检索注入 | 备选「手动塞 prompt」放弃：用户负担重，无法弹性扩展知识量 |
| 3 | 澄清前置 | 接单先生成澄清卡片，**用户确认后才动手** | 备选「直接执行 + 中途纠偏」放弃：返工成本高，体验割裂 |
| 4 | 双向沉淀触发 | 每个需求完成/驳回都触发沉淀，**不需用户主动归档** | 备选「用户审核后写入」放弃：摩擦大，长期沉淀率会归零 |

# 第二部分：核心概念与数据模型

## 五、四大核心实体 + 一种沉淀产物

```
┌──────────┐      拥有       ┌──────────┐
│  员工     │ ─────────────▶ │ 技能      │
│ Employee │                │  Skill   │
│ + memory │                │ + prompt │  (含岗位通用规范)
└────┬─────┘                └──────────┘
     │ 接单
     ▼
┌──────────┐    1:1      ┌──────────┐
│  需求     │ ──────────▶│ Thread   │
│Requirement│             │+ Messages│  (思维链事件流)
└────┬─────┘             └──────────┘
     │ 归属
     ▼
┌─────────────────────────────────────┐
│            项目 Project              │
│ knowledge {                          │
│   facts       (软提示 RAG)           │
│   pitfalls    (软提示 RAG)           │
│   conventions (硬约束 required/RAG)  │
│ }                                    │
└─────────────────────────────────────┘
     ▲
     │ 完成/驳回 → 自动产出 Report
     │           分流写入：
     │   ├──→ project.knowledge.{facts,pitfalls,conventions}
     │   └──→ employee.memory.{style,lessons}
     │
┌──────────┐
│  汇报     │
│  Report  │
└──────────┘
```

**核心关系**：
- `员工 → 技能 → 接需求 → 归属项目`
- 每个需求 1:1 关联 1 个 Thread；Thread 内是 Message 事件流（思考、工具调用、澄清问答）
- 需求完成/驳回后产出 Report，按主题分流沉淀到 **项目知识库（含 facts / pitfalls / conventions 三类）** 和 **员工记忆（style + lessons）**

### 5.1 员工（Employee）
| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid | |
| name | string | 员工昵称，如「小李 - 前端」 |
| avatar | string | 头像 |
| role / 岗位 | enum/自定义 | 例：前端工程师、产品经理、文案 |
| persona | text | **用户初始手写**的角色设定（不可变，除非用户主动改） |
| skills | Skill[] | 关联技能 |
| model_config | json | 使用的 LLM 模型、温度、最大 token；含用户自带 Key（**加密存储**） |
| tool_access | string[] | 可调用的工具列表（员工层硬上限，详见 §5.2） |
| memory | json | `{ style: text, lessons: MemoryItem[] }`；**style 由系统从历史复盘自动追加更新**，与 persona 不同 |
| status | enum | `active` / `archived` |
| stats | json | 已完成需求数、平均耗时、成功率 |
| reports | Report[] | 关联的需求总结 |
| created_at | datetime | 入职时间 |

> **persona vs memory.style**：persona 是用户写的"我希望这个员工是什么风格"（输入），style 是系统反思后总结的"他实际表现出什么风格 + 偏好"（沉淀产物）。两者都注入 prompt，但来源和可变性不同。

### 5.2 技能（Skill）
| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid | |
| name | string | 例：React 开发、SEO 文案、用户访谈 |
| category | enum | 技术 / 设计 / 内容 / 数据 / 运营… |
| description | text | 技能能做什么 |
| prompt_template | text | 调用此技能时注入的提示模板 |
| required_tools | string[] | 此技能依赖的工具 |
| examples | json | 示例输入输出，用于 few-shot |

> 设计要点：**技能是可复用单元**，多个员工可拥有同一技能；技能的 prompt + tools 组合定义了员工的真实能力边界。

#### 多技能 prompt 合并规则
一个需求可挂多个技能。注入顺序：
1. 员工技能列表的**第一个技能视为主技能**，其 `prompt_template` 完整注入
2. 其余技能仅注入 `name + description`，作为"额外可用能力"提示
3. 所有挂载技能的 `required_tools` 取**并集**作为本次任务的工具候选集

#### tool_access 与 required_tools 冲突
- 员工 `tool_access` 是**硬上限**：未授权的工具，技能也无法调用
- 技能 `required_tools` 中**未被员工授权**的工具：**降级为不调用**，思维链中标注"⚠️ 技能要求 X 工具但未授权"
- 派单时若主技能的核心工具未被授权，系统在澄清卡片中提示用户"建议为该员工授权 X 工具，否则结果可能受限"

| 字段 | 类型 | 说明 |

### 5.6 会话线程（Thread）与消息（Message）

> 解决 `Requirement.chat_thread` 字段悬空问题：明确执行过程中的对话存储模型。

**Thread**：一个需求的整个执行过程对应 **1 条 Thread**（一对一）。Thread 是逻辑容器，不跨需求复用。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid | |
| requirement_id | uuid | 一对一 |
| created_at | datetime | |

**Message**：Thread 下的消息事件流，按时间单调递增：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid | |
| thread_id | uuid | |
| seq | int | 单 Thread 内单调递增 |
| role | enum | `system` / `user` / `assistant` / `tool` |
| type | enum | `text` / `thinking` / `tool_call` / `tool_result` / `clarification_request` / `clarification_answer` / `plan_update` / `error` |
| content | json | 消息正文（按 type 不同结构不同） |
| tokens | json | `{ input, output, cached }` 用于成本统计 |
| created_at | datetime | |

> **跨需求历史**：V1.0 **不串联**。同员工接下一个需求时，前一个需求的 Thread 不进入新 prompt——避免"上一个需求的临时上下文污染下一个"。跨需求的经验沉淀走 §六 记忆模型（facts/pitfalls/style/lessons），不走 Thread。

### 5.7 澄清（Clarification）
> **每次中断生成一条 Clarification**（接单为第 0 轮；执行中每次员工主动暂停发问为新一轮，round 单调递增）。一次澄清的多个问题放在 `questions` 数组里。
> **与 Requirement.plan 的关系**：接单时 Clarification.proposed_plan 经用户确认后**初始化** Requirement.plan.steps；执行中用户可继续编辑 plan（§5.3 改计划），不再回写 Clarification。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid | |
| requirement_id | uuid | |
| round | int | 接单时=0，执行中每次中断 +1 |
| trigger | enum | initial(接单) / decision_split / missing_info / pitfall_hit / cost_alert / judgment |
| employee_understanding | text | 员工的理解复述（仅接单轮填写） |
| proposed_plan | text[] | 拆解步骤（仅接单轮填写） |
| questions | json[] | `[{ question, answer?, answer_mode }]`；`answer_mode`: `user`(等用户答) / `auto_proceed`(用户跳过、员工按理解干) |
| resolved_at | datetime? | |

## 六、记忆与规范模型（分层）

**决策**：业务知识沉淀**主归项目**，员工只保留极轻量的"自我画像 + 个人教训"。

> "员工是带着技能来项目里干活的人，知识属于项目，人属于团队。"
> 员工换项目不应该带走项目的业务知识；项目换员工不应该丢失业务积累。

### 6.1 记忆分层（含规范）

V1.0 把"员工带进项目的知识"分为**四层**，按强制程度和归属各有不同：

| 记忆层 | 字段位置 | 角色 | 存什么 | 注入策略 |
|---|---|---|---|---|
| **岗位通用规范** | `skill.prompt_template` | 硬约束 | React Hooks 用法、Go error 处理惯例等**岗位/技能层通用规则** | 该技能被挂载时**全量注入** |
| **项目规范** | `project.knowledge.conventions` | 硬约束 | "本项目用 Zustand 而非 Redux"、API 走 tRPC、命名规则等**项目特定规则** | `required` 条目**全量注入**；`recommended` 条目 **RAG 检索 Top-K 注入** |
| **项目知识 + 踩坑库** | `project.knowledge.facts` + `project.knowledge.pitfalls` | 软提示 | 项目背景、术语、技术栈、决策、关键文档、**本项目踩过的坑** | 需求归属此项目时**自动 RAG 检索 Top-K 注入** |
| **员工自我画像 + 教训** | `employee.memory.style` + `employee.memory.lessons` | 辅助 | 工作风格、偏好、**个人踩坑教训** — 不存业务知识 | style 全量注入；lessons 按相关性 Top-K 注入 |
| **需求上下文** | `requirement.description` + 附件 | 临时 | 本次任务的具体描述 | 单次执行，不沉淀 |

> **规范 vs 知识**：规范是"违反就是错"的硬约束（命中频次接近 100%）；知识是"相关时参考"的软提示（按场景 RAG）。两者注入策略和管理方式都不同。

#### 记忆条目（MemoryItem）最小结构
facts / pitfalls / lessons 都是条目数组，单条结构统一：
```
{
  id: uuid
  content: string         // 条目正文
  source_requirement_id?: uuid   // 来源需求（自动生成的条目带，手动新增可空）
  hit_count: int          // 被检索命中次数
  last_hit_at?: datetime
  importance_score: float // 0.0 ~ 1.0；详见下文 Importance Scoring
  user_feedback: enum     // none / positive(用户验收时显式认可) / negative(被驳回时关联)
  created_at: datetime
  archived: bool          // 低分长期未命中或用户手动归档
}
```
`employee.memory.style` 是单段 text（非数组），允许全量覆盖。

#### Importance Scoring（重要性评分）
单纯 LRU 在条目过千时不够用——必须有"分数化"。V1.0 采用**简化打分公式**（不上 RL）：

```
importance_score = 0.4 × normalize(hit_count)
                 + 0.3 × user_feedback_weight        // positive=1, none=0.5, negative=0
                 + 0.2 × recency(last_hit_at)        // 越近越高
                 + 0.1 × source_quality              // 用户手动新增=1, 系统自动=0.7
```

- 检索时按 `相似度 × importance_score` 综合排序
- 每周自动批处理：`importance_score < 0.2` 且 30 天未命中 → 自动归档
- 用户在 UI 可看到每条记忆的当前分数（便于调试）

#### 分类置信度（防错分类）
自动复盘把内容分流到 facts / pitfalls / lessons / style / convention 时，**LLM 同时输出置信度** `0.0~1.0`：

- 置信度 ≥ 0.8：直接写入对应字段
- 置信度 0.5 ~ 0.8：写入 `pending_review` 暂存区，需求详情页展示"待确认沉淀（3 条）"，用户一键确认或调整分类
- 置信度 < 0.5：**不沉淀**，思维链记录"⚠️ 本条复盘内容分类不确定，未写入记忆"

这是降低"垃圾沉淀污染记忆库"风险的最低门槛。

#### 规范条目（ConventionItem）结构
项目规范条目独立结构，多一个强制等级字段：
```
{
  id: uuid
  content: string         // 规范正文（Markdown，单条建议 < 500 token）
  enforcement: enum       // required(全量注入) / recommended(RAG Top-K)
  category?: string       // 自由分类，如 "命名" / "目录结构" / "依赖管理"
  created_at: datetime
  updated_at: datetime
}
```

> **关于跨项目复用**：V1.0 每个项目独立维护规范，**不做跨项目共享**。常见模式是：用户先在一个项目里写好前端规范，新建相似项目时手动复制粘贴。规范库实体推迟到 V1.1+ 视使用反馈再上。
>
> **关于长规范**：单条建议 < 500 token；超长规范用户应**自行拆分为多个条目**，并合理标注 `required` / `recommended`（required 全量注入会显著推高 token 成本）。V1.0 不做长文档自动分段。

### 6.2 双向沉淀（每个需求总结都触发）

需求完成或被驳回后，引擎自动从 Report.content 中按主题分流写入：

```
需求总结（无论验收还是驳回）
   │
   ├─ 业务事实 / 决策        ──→ project.knowledge.facts
   ├─ 项目踩坑              ──→ project.knowledge.pitfalls
   ├─ 项目规范候选           ──→ project.knowledge.conventions (pending_review)
   ├─ 工作风格反思           ──→ employee.memory.style
   └─ 个人教训              ──→ employee.memory.lessons
```

> 分流由 LLM 完成，**配合分类置信度机制**（详见 §6.1）：高置信直接写入，中置信进 pending_review，低置信不沉淀。

### 6.3 纠错触发条件
任一即触发自动复盘 + 写入教训库：
- 需求被**驳回**（status: 已驳回）
- 需求被**多次返工**（≥2 次修改才验收）
- 用户在交付物 / 验收时给**负面反馈**（关键词检测 + LLM 判定）

### 6.4 教训分流（驳回 / 返工 / 负反馈）
```
复盘
  ├── 我个人应该改的     → employee.memory.lessons
  │   "我容易没确认就动手，下次先问"
  │
  └── 这个项目的踩坑     → project.knowledge.pitfalls
       "客户讨厌长邮件，要求 200 字以内"
```

教训写入时携带 `user_feedback: negative`，进入 Importance Scoring 公式（§6.1）。

### 6.5 自动注入（新需求接单时）
按强制等级从硬到软排列，规范在前、知识在后、需求描述压轴：
```
[员工 persona]                        ← 用户初始人设
[员工 memory.style]                   ← 系统沉淀的工作风格
[主技能 prompt_template]              ← 含岗位通用规范（全量）
[项目 conventions - required]         ← 项目硬规范（全量）
[项目 conventions - recommended]      ← 项目软规范（RAG Top-K）
[项目 facts]                          ← 项目知识（RAG Top-K）
[项目 pitfalls]                       ← 项目踩坑（RAG Top-K）
[员工 lessons]                        ← 个人教训（RAG Top-K）
[需求描述]
```
> 硬约束放在 prompt 前部，LLM 注意力更强；软提示通过 RAG 按需提供。

### 6.6 用户可干预
- 员工详情页「记忆」Tab：可手动编辑 style / 增删 lessons
- 项目详情页「项目知识」Tab：可手动增删 facts / pitfalls
- 项目详情页「**项目规范**」Tab：可手动增删 conventions、标注 required/recommended
- pending_review 暂存区在需求详情页底部显式提示，用户一键确认 / 调整分类 / 丢弃

### 6.7 避免无限累积
- 同主题条目自动去重 + 合并（基于 embedding 相似度 ≥ 0.92 触发 LLM 二次判定）
- Importance Scoring 周批处理：低分 + 长期未命中 → 自动归档（详见 §6.1）
- 用户可手动清除

### 6.8 跨需求 Recall 边界（重要约束）
- **同员工跨需求**：上一个需求的 Thread / Message **不**进入下一个需求的 prompt
- 跨需求传递只通过 §六 沉淀机制（facts / pitfalls / style / lessons）
- 设计动机：避免临时上下文污染长期记忆；让员工的"经验"和"对话"彻底分离

### 6.9 形态层面
- **项目详情页**含「项目知识」+「项目规范」两个 Tab：可浏览、编辑、手动追加条目
- **员工详情页**的「记忆」Tab 展示自我画像 + 个人教训
- 需求执行时上下文拼装顺序见 §6.5
- 项目归档：`project.knowledge` 转为只读历史；新建相似项目时可一键克隆作为起点
- 员工 archive：员工记忆是"人的画像"，可跨项目复用；项目知识不带走（业务保密 + 避免串味）

# 第三部分：V1.0 — 引擎验证版

> **定位**：纯引擎。验证"AI 员工能不能**接活 → 澄清 → 执行 → 沉淀 → 下次更好**"这个核心闭环。
> **不是产品，是引擎**。所有商业化、运营、协作、生态能力一律推迟。
> **周期**：两阶段（α 4 周 demo + β 4~8 周 stable，详见 §三）

## 九、V1.0 定位与边界

### 9.1 想验证什么
只验证**一个问题**：

> 让一个配置过技能和人设的 AI 员工，接到一个项目内的需求，能不能在「澄清前置 + 过程透明 + 记忆沉淀 + 纠错学习」这套机制下完成任务，并且下次同类任务做得更好？

跑通了，产品方向被验证；跑不通，就重新设计，不浪费时间在仪表盘、计费、模板市场这些"运营外壳"上。

### 9.2 单用户本地使用
- 不做用户系统 / 登录 / 多租户 / 多人协作
- 所有数据都是当前操作者的
- 用户**自带 LLM Key**（在员工配置中填入），api_key **加密存储**
- 平台不收费、不限速、不计量

### 9.3 执行模型与持久化
- **同步执行**：派单后由前端发起，引擎持续运行直到完成 / 等待回答 / 暂停 / 失败
- **状态实时落库**：思维链每一帧、当前步骤、下一步计划在产生时立即写入数据库；**刷新页面 / 重新打开应用后能从落库状态恢复**，看到执行到一半的需求继续滚动
- **多需求并发**：同一时刻允许多个需求处于"进行中 / 等待回答"状态；用户在需求列表里可来回切换查看
- **意外中断兜底**：进程崩溃 / 网络断开后，需求状态保留为"进行中"；用户重新打开时引擎检测到无活跃 worker，提示用户选择"继续执行"或"标记为已暂停"
- **LLM API 失败**：失败时需求转入"已暂停"，思维链记录错误信息；用户可选"重试 / 修改提示 / 强制结束"
- **Prompt Cache**：引擎在拼装 prompt 时，前缀（persona + memory.style + 主技能 prompt + required conventions）保持稳定，启用 LLM provider 的 prompt cache（Anthropic / OpenAI 均支持），输入 token 成本可降至 ~10%

### 9.4 执行 Budget Cap（防失控）
> 行业经验：LLM agent 死循环（思考→工具→反思→思考……）是单需求"烧 Key 灾难"的头号原因。V1.0 必须有硬上限。

每个需求都强制三道闸门，任一触发立即中断转入"已暂停"：

| 闸门 | 默认值 | 来源 |
|---|---|---|
| `max_iterations` | 30 轮（思考+工具）循环 | 系统默认，员工/项目可单独调整 |
| `max_tokens` | 200,000 token 累计输入+输出 | 同上 |
| `max_wall_time` | 30 分钟实际墙钟时间 | 同上 |

触达任一阈值时：
- 思维链记录"⏹ 触达 X 阈值，需求自动暂停"
- 需求状态 → `已暂停`
- 用户可选"继续（增加 budget）/ 强制结束 / 丢弃"

### 9.5 Context Length 超限处理
单需求执行中累计 prompt token 接近模型上限时（默认 80% 阈值）：
1. 引擎对**早期思维链**做自动摘要（保留首尾 + LLM 压缩中间段为 1-2 段）
2. 摘要后的思维链在前端显示为可折叠"… 已摘要 N 帧 …"
3. 项目知识 / 教训等 RAG 注入条目优先级排序保留 Top-K，超出部分截断
4. 摘要本身也消耗 token，计入 budget cap

### 9.6 工具调用错误处理（行业头号失败原因）

> G2 2026 调研：6/7 头部 agent 厂商认为"API/Integration failures"是 agent 失败的最大原因，不是 LLM 本身。这一节是 V1.0 工程级稳定性的关键。

每次工具调用都过以下闸门：

| 类别 | 处理策略 |
|---|---|
| **超时** | 单次工具调用默认 30s 超时；超时后**指数退避重试** 2 次（30s → 60s → 120s）；仍失败则记 `tool_result` 类型 `error`，LLM 看到错误自决是否换方案 |
| **返回非预期格式** | 工具响应做 JSON Schema 校验，校验失败视同错误返回；不直接让 LLM 看到污染数据 |
| **死循环检测** | 检测"同工具 + 同参数 ≥ 3 次"或"同两个工具反复来回 ≥ 5 次"→ 强制注入提示"⚠️ 检测到工具调用循环，请改变策略或调用 `ask_user`" |
| **未授权工具** | LLM 调用 `tool_access` 外的工具 → 直接返回错误，不实际执行；思维链显示"⚠️ 工具 X 未授权" |
| **Budget 计入** | 每次工具调用（含失败重试）的 token 和墙钟时间都计入需求 budget cap（§9.4） |
| **致命错误** | 连续 ≥ 3 个工具调用全失败 → 需求转入"已暂停"，提示用户介入 |

### 9.7 同步执行 + 多需求并发的工程难度声明

V1.0 在浏览器中支持多需求并发同步执行——这是**最难做对的部分**，工程难点：

- 多个 LLM stream 同时进行：SSE 多路复用 + 状态广播 + IndexedDB 落盘
- 同一 LLM Key 的 rate limit 共享：需要全局调度器排队
- 关闭某个 tab 时：其余需求继续执行，关闭的需求**前端断开**但**引擎继续跑**直到完成或 budget cap
- 浏览器整体关闭：所有"进行中"需求在重新打开时进入 §9.3 "意外中断兜底"流程

> α 阶段可降级为"同一时刻只允许 1 个需求活跃执行"（其余排队），β 阶段再放开真并发。这是已知的工程妥协项。

> 完整的 ❌ 不做清单见 §11；模块优先级与降级方案见 §13。

## 十、V1.0 功能模块（十个 M）

### M1. 项目（知识与规范容器）
- **列表**：项目名 + 简介 + 需求数
- **详情**：
  - 介绍编辑（富文本/Markdown）— 保存即触发向量化
  - 「**项目规范**」Tab：conventions 列表
    - Markdown 编辑器，每条规范可标注 `required` / `recommended`
    - 自由分类（命名 / 目录结构 / 依赖管理 / 代码风格 / ...）
    - 编辑后立即生效，下次派单的 prompt 拼装直接采用
    - 典型示例：
      - 前端项目：技术栈、组件命名、状态管理、目录结构、API 调用约定
      - 后端项目：服务端依赖白名单、错误处理惯例、日志规范、数据库变更流程
  - 「项目知识」Tab：facts 列表 + pitfalls 列表（均可手动编辑/删除）
  - 「需求」Tab：本项目下的所有需求
- **自动 RAG**：员工接需求时按需求描述检索 Top-K 注入（facts / pitfalls / recommended conventions）
- **规范文件兼容（关键）**：开发型项目可绑定 git 仓库路径，**自动读取以下文件作为 conventions 来源**：
  - `AGENTS.md`（2026 开放标准，60k+ 仓库已采用）
  - `CLAUDE.md`（Claude Code 约定）
  - `.cursorrules` / `.cursor/rules/*.md`（Cursor 约定）
  - 按 H2 章节自动切分为 ConventionItem 条目，默认 `enforcement=required`
  - **同步策略**：UI 展示为只读 + "在编辑器中打开"按钮，避免双向同步冲突；文件变更后引擎在下次派单时重新读取
  - 用户若在 UI 内新增的条目存入 `project.knowledge.conventions`（不写回文件），与从文件读取的条目并列展示但分组区分
- **规范跨项目复用**：V1.0 不做。用户在新建相似项目时可手动复制粘贴规范条目；或者把规范沉淀到仓库的 AGENTS.md 里，新项目复用同一份仓库即可（推迟到 V1.1+ 视反馈做规范库实体）
- **Embedding 依赖**：
  - 默认调用 OpenAI `text-embedding-3-small`（用户在「系统设置」中单独配置 embedding API Key）
  - 若用户只有 Anthropic Key：兜底用本地嵌入模型（如 `bge-small`）跑在本地
  - Embedding Key 缺失时，向量化按需失败但**不阻断需求执行**：员工接需求时若无项目知识可检索，思维链中提示"项目知识库未启用"；required conventions 由于全量注入，**不依赖 embedding 仍生效**

### M2. 员工（能力载体）
- **列表**：头像 + 名字 + 岗位 + 技能标签 + 状态
- **详情**：
  - 基本信息（名字、岗位、人设/persona）
  - 技能挂载（多选）

- **临时对话** ⚠️ P2，可延后到 V1.1：员工详情页内置对话面板，绕开需求流程直接聊（用于调试人设）
- ⚠️ V1.0 **不做**岗位模板系统，员工通过手动配置创建

### M3. 技能（能力单元）
- **列表**：name + category + description
- **编辑器**：name / category / description / prompt_template / required_tools / examples
- **系统预置（8 个起步技能，α 阶段最少 5 个）**：
  - 通用：通用对话、文档总结、联网搜索 *(α 必带)*
  - 内容：长文写作、营销文案 *(α 必带 1 项)*、用户访谈整理 *(β)*
  - 开发：代码生成与解释、需求拆解 *(β)*
- **预置技能定位**：仅为"脚手架样例"，让用户上手时能立即派一个需求跑通。**V1.0 不追求覆盖真实工作所需的全部技能**——用户应基于自己的场景创建自定义技能。
- **岗位模板**：V1.0 不做（推迟到 V2.0）

### M4. 需求（任务入口）
- **列表**：表格视图（标题 / 项目 / 员工 / 状态 / 优先级 / 累计 token），按状态筛选
- **新建**：
  - 必填：标题、描述、项目、员工
  - 选填：所需技能、优先级
  - V1.0 仅支持「**同步执行**」
- **详情**：澄清卡片 → 思维链三栏 → 交付物 → 验收/驳回（详见 M5/M6）
- **成本与 Budget 可见**：需求详情页底部状态条始终显示：
  - 累计 token（input / output 分开）+ 估算美元成本（按模型公开价 × 用量）
  - 迭代进度 `N/max_iterations`、耗时进度 `t/max_wall_time`
  - 触达任一 budget 阈值时弹出"继续 / 强制结束 / 丢弃"
- ⚠️ V1.0 **不做**看板视图（列表足够）

### M5. 核心引擎（V1.0 的灵魂）

#### 适合 / 不适合的需求类型（派单前 anti-pattern 提示）
> 真实使用中 70% 失败案例源于"派了不该派给 AI 员工的需求"。新建需求页面给出明确提示：

**适合派**：
- 内容创作（文案 / 文档 / 报告 / 长文）
- 方案分析 / 拆解 / 评估
- 单文件代码片段（不依赖跨文件理解）
- 数据整理 / 文档总结
- 用户访谈整理 / 调研报告

**不适合派给 V1.0 员工**：
- ❌ 涉及外部账号操作（登录、发邮件、改 CRM）—— V1.0 无 OAuth / integration
- ❌ 需要实时交互决策（直播、客服快速响应）—— 同步执行延迟不可控
- ❌ 涉及物理动作或浏览器 UI 操作 —— 无 Computer Use 能力
- ❌ 跨多文件理解的大型代码重构 —— 无完整 codebase 索引
- ❌ Git/PR/CI 闭环（自动创建 PR / 修 CI）—— V1.0 交付物 = 文件下载，不接 git
- ❌ 多模态输入（图像理解 / 语音）—— V1.0 纯文本 + 文件

新建需求页面在用户填描述时**实时检测关键词**（如"发邮件"、"登录"、"PR"、"截图分析"），命中即提示"这类需求 V1.0 员工可能无法独立完成"，并引导用户细化或改派。


#### 5.1 接需求 → 澄清卡片（带复杂度阈值）
员工接到需求后先评估复杂度，**非简单需求才生成澄清卡片**。设计动机：
> 强制所有需求走澄清会让用户烦躁（行业反方向是"先试错再修"）。复杂度阈值保留了"宁可多问"在真正必要时的价值，又避免在简单任务上消耗用户耐心。

**判定流程**：
1. LLM 先输出一段"复杂度自评"：`simple / standard / complex`，并给出理由（≤ 50 字）
2. 判定规则：
   - `simple`（如"帮我把这段中文翻译成英文"）→ **跳过澄清直接执行**，复杂度自评仍写入思维链开头
   - `standard / complex` → 走完整澄清卡片流程（理解复述 + 拆解步骤 + 3~5 个问题）
3. 用户**可全局开关**："总是澄清 / 智能判断（默认） / 总是跳过"——在系统设置中调整
4. 命中项目 pitfalls / 关键工具未授权时**强制澄清**，不受阈值影响

需求状态：
- 简单需求：`待分派` → `进行中`（无 `待澄清` 中间态）
- 复杂需求：`待分派` → `待澄清` → 用户确认/回答后 → `进行中`

#### 5.2 执行：思维链三栏
左栏思维链流式 / 中栏当前步骤 / 右栏下一步计划（用户可编辑）。

#### 5.3 用户控制权
- ⏸ **暂停**：立即停下，可追加说明后继续
- 💬 **追问**：在思维链上插入消息
- ✏️ **改计划**：直接编辑右侧步骤列表
- ⏹ **强制结束**：终止，可选"保留产出 + 验收"或"丢弃"

#### 5.4 执行中再澄清
5 种触发场景员工**主动暂停发问**（V1.0 必做 2 种，其余按 §十三 优先级降级）：

| 触发场景 | 员工行为 | V1.0 |
|---|---|---|
| 方案分歧 A vs B | 列选项 + 利弊 → 询问 | **P0-α** |
| 关键信息缺失 | 说明需要什么 → 等待 | **P0-α** |
| 判定不准 | 给 2-3 个变体 → 询问偏好 | P0-β |
| 命中项目 pitfalls | "上次这里出过问题，这次想这样做" → 确认 | P0-β |
| `cost_alert`（LLM 预估接下来会超 budget） | 说明预估 → 询问是否继续 | P0-β |

> **cost_alert vs Budget Cap 区分**：
> - `cost_alert`（本表第 5 行，软提示）：LLM **预估**接下来的步骤会触达 budget，主动暂停发问"还要继续吗"——给用户**预先决策权**
> - `Budget Cap`（§9.4，硬阈值）：引擎**实际**触达 max_iterations / max_tokens / max_wall_time，**强制中断**——是兜底安全网
>
> cost_alert 是 LLM 自评机制（可被 prompt 调教，准确率有限）；Budget Cap 是引擎硬规则（100% 可靠）。两者互补：cost_alert 提供柔性体验，Budget Cap 保证底线。

> **触发机制**：将"提问"实现为**一个系统级工具 `ask_user(questions, trigger_reason)`**，作为 LLM 可见工具的一部分注入。LLM 想中断发问时**必须显式调用此工具**——引擎收到 tool call 立即中断执行、生成 Clarification 记录、转入"等待回答"状态。
>
> 这种"工具调用驱动"比纯 prompt 指令的文本标签可靠一个数量级，是行业 2026 年主流稳定方案。

需求状态：`进行中` → `等待回答` → 用户回复 → `进行中`。

#### 5.5 完成 → 自动总结
员工产出最终交付物，用户验收 / 驳回。完成后自动生成单需求总结，按 M6 规则分流写入记忆。

### M6. 记忆与纠错引擎

> 完整设计见 **§六 记忆与规范模型**。本节仅声明引擎在该模型下的实现职责，不再重复内容。

引擎在需求生命周期中执行三件事：
- **完成 / 驳回时**：按 §6.2 触发双向沉淀流程，按 §6.4 处理纠错路径
- **接单时**：按 §6.5 自动注入顺序拼装 prompt（硬约束在前、软提示在后）
- **后台批处理**：按 §6.7 周期性运行去重合并 + 自动归档（依赖 §6.1 Importance Scoring）

### M7. 工具注册地基（V1.0 必做，但内容极简）
> 即使 V1.0 不做 Integration 生态，**统一工具注册机制必须先打地基**，否则 V1.1 加任何工具都要重构。

- **Tool 实体**：`{ id, name, description, input_schema(JSON Schema), invoke_fn_ref, requires_auth }`
- **统一接口**：所有工具（含 `ask_user` 在内）通过同一 registry 注册，LLM 看到的是统一 tool_call 协议
- **V1.0 自带工具**：
  - `ask_user`（再澄清，§5.4）
  - `web_search`（联网搜索，可选）
  - `read_file` / `write_file`（本地文件读写，沙箱限定在用户指定目录）
- **不做**：OAuth、第三方 SaaS 集成、MCP 协议接入（V1.1+）

### M8. Onboarding 样板项目（防 90% 用户在第 3 步流失）
首次启动应用，引导用户走"加载样板 / 跳过"二选一：

- **加载样板**：一键导入 3 个示例项目（前端 Web 应用 / 内容运营 / 个人写作）+ 5 个示例员工（含完整 persona、技能挂载、若干历史需求与已沉淀记忆）
- 样板项目带 mock LLM Key 占位，用户用前先替换为自己的 Key
- 用户在样板里可以**立即看到** facts / pitfalls / conventions / 思维链 等所有能力的实际形态

**两档实现方案，β 阶段选其一**：

| 档位 | 说明 | 工程量 |
|---|---|---|
| **Lite（默认）** | 样板需求展示为**已完成态截图**（思维链 + 交付物的静态快照） | 低，1 周 |
| **Pro** | 思维链回放器：从 Message 表读静态数据 + 模拟流式延迟 + 跳过工具实际调用 | 高，~3 周 |

> Lite 已足够覆盖"让用户立即看到能力形态"目标，是 β 阶段默认实现；Pro 视用户调研结果决定是否上。

**首次填 Key 引导**（与 M2 联动）：
- Onboarding 第 2 步：引导用户填一个全局 LLM Key（替换 mock 占位）
- 派任何真实需求前检测员工 Key，若缺失：弹窗"请先在员工详情页填入 LLM Key"，跳转到员工配置页（不阻断用户浏览样板）

### M9. Replay & Debug（最小可用版）
- 任意已完成 / 已驳回需求详情页提供「**Replay**」按钮：用相同输入 + 当前的 prompt 模板 + 当前的记忆库重新跑一次
- Replay 结果与原结果**并排展示**，方便用户对比"改了 prompt / 加了规范后效果有没有变好"
- Replay 不写回任何记忆库（不污染历史）
- V1.0 不做：批量 replay、LLM-as-judge 自动评分、A/B 框架（V1.1+）

### M10. Governance（数据与凭证治理）
单用户本地虽风险低，仍需明确数据边界：

| 项 | V1.0 处理 |
|---|---|
| LLM Key 存储 | 操作系统级凭证管理（macOS 钥匙串 / Windows DPAPI / Linux Secret Service 等）；不落地数据库明文 |
| 数据存储位置 | 本地嵌入式数据库 + 用户文档目录可见路径，用户可备份可迁移；具体技术选型见技术设计文档 |
| 项目删除 | **级联删除**所有需求 / Thread / Message / Report / project.knowledge；二次确认 + 30 秒撤销窗口 |
| 员工删除 | **保留**该员工历史需求与 Report；员工本身 archive；持有的 lessons 标记 orphan 但不删 |
| 工具授权撤销 | 撤销 `tool_access` 时，正在执行的需求**立即收到错误**（下一次 tool_call 时返回未授权），由 LLM 自决暂停或换方案 |
| 数据导出 | 单个需求的交付物 + Thread 可导出；全量数据导出 V1.1 |

## 十一、V1.0 明确不做（重要）

为保持引擎纯净，以下能力**明确不做**：

### 商业化与多人协作
- ❌ 用户系统、登录、注册、找回密码
- ❌ 多用户、多团队、权限、共享
- ❌ 付费、充值、余额、账单、计量

### 仪表盘与汇报
- ❌ 仪表盘、数据看板、统计报表
- ❌ 日报 / 周报 / 月报

### 协作与扩展
- ❌ 模板市场、岗位模板分享、**跨项目共享 Brain**（已知不足，V1.1+ 视反馈考虑）
- ❌ 员工经理、员工间转派、协作请教
- ❌ 自动技能匹配、员工推荐算法

### 自动化触发与集成
- ❌ **Trigger / 事件驱动入口**（邮件/Slack/定时器触发员工自主工作）—— 行业主流（Lindy/Sintra）的核心能力，V2.0 才考虑
- ❌ **Integration 生态**（Gmail / Slack / HubSpot / Calendar 等第三方 SaaS 接入），V1.0 仅打工具注册地基（M7）
- ❌ MCP 协议接入（V1.1+）
- ❌ 异步执行、后台任务、定时任务
- ❌ 站内通知中心、邮件、短信、微信推送

### 开发型员工的"真活"边界
- ❌ **完整 codebase 索引 / AST 解析 / 跨文件理解**（Devin / Cursor 路线），V1.0 开发员工只能产出**方案 / 文档 / 单文件代码片段**
- ❌ **Git 集成 / PR 创建 / CI 反馈循环 / 测试自修复**（Devin 路线），交付物 = 文件下载，V1.1+ 再考虑
- ❌ Computer Use / 浏览器 UI 操作（Lindy Autopilot / Anthropic Computer Use 路线），定位差异

### 评估与质量
- ❌ 批量 Replay、LLM-as-judge 自动评分、A/B 框架（M9 仅支持单次手动 replay）
- ❌ 多模态（图像 / 语音输入输出），V1.0 纯文本 + 文件

### 其他
- ❌ 移动端
- ❌ 全量数据导出（V1.0 仅支持单需求交付物 + Thread 导出）
- ❌ 国际化 / 多语言（仅中文）
- ❌ 主题切换、深色模式

如发现以上任何一项是"必须做"的，走变更评审，不能直接加。

## 十二、V1.0 验收标准（端到端场景）

### 主路径（功能完整性）
1. **基础配置**：创建项目（含介绍）→ 创建员工（挂技能、填 LLM Key、写人设）→ 项目介绍向量化完成可查询
2. **澄清前置跑通**：派需求 → 员工生成澄清卡片 → 用户确认 → 员工开始执行
3. **思维链透明**：执行过程中能看到思考、工具调用、当前步骤、下一步计划；用户可暂停、追问、改计划
4. **执行中再澄清**：构造方案分歧场景 → 员工暂停发问 → 用户回答 → 员工继续
5. **记忆沉淀可见**：完成需求 → 项目知识库出现新 facts → 员工 style 有更新
6. **纠错学习闭环**：驳回需求 → 教训写入 lessons + pitfalls → 派同类型新需求 → 在澄清卡片或思维链中能**显式引用**至少 1 条历史教训

### 持久化与并发
7. **会话恢复**：执行中刷新页面 / 重启应用 → 能看到需求继续滚动，思维链历史完整
8. **多需求并发**：同时派 2 个需求给不同员工 → 两个需求列表项都能独立进入"进行中"，互不阻塞

### 异常兜底
9. **LLM 失败**：故意切断网络 / 用错 Key → 需求转入"已暂停"，思维链记录错误，可重试
10. **用户取消**：在"待澄清"状态下取消需求 → 状态变为"已取消"，不触发任何记忆沉淀
11. **Embedding 缺失**：未配置 embedding Key 且无本地兜底模型时 → 需求执行不阻断，思维链提示"项目知识库未启用"

### 规范遵守
12. **required 规范命中**：在项目规范中加一条 required 条目（如"所有 API 必须用 tRPC，禁用 fetch"）→ 派一个会触发该规范的需求 → 员工在澄清卡片或思维链中**明确遵守该规范**（如选 tRPC 而非 fetch）
13. **recommended 规范柔性**：加一条 recommended 条目 → 派需求 → 员工**在相关场景**才引用，无关场景不强行带入

### Budget 与成本可见
14. **Budget 触达**：故意把 `max_iterations` 调到 3 派一个需要更多迭代的需求 → 触达后需求自动转入"已暂停"，思维链记录原因，用户可一键"继续（增加 budget）"
15. **成本可见**：执行中需求详情页底部实时显示 input/output token + 估算美元成本，迭代/时间进度条同步更新
16. **复杂度阈值**：派一个简单需求（如"翻译这段话"）→ **不出现澄清卡片**，直接进入执行；派一个复杂需求 → 出澄清卡片
17. **`ask_user` 工具调用**：构造方案分歧场景 → 员工通过 `ask_user` 工具中断 → 引擎正确转入"等待回答"状态（验证机制是工具调用而非文本标签）

### 工具与失败处理
18. **工具超时重试**：mock 一个会超时的工具 → 引擎按指数退避重试 2 次 → 最终失败时返回错误给 LLM，LLM 能自决换方案或调用 `ask_user`
19. **死循环检测**：让员工故意反复调同一工具同参数 ≥ 3 次 → 引擎注入循环检测提示并强制中断
20. **未授权工具**：撤销员工某工具授权后，LLM 仍尝试调用 → 直接返回未授权错误，不实际执行

### 记忆质量
21. **Importance Scoring**：完成多个需求 + 用户给若干验收/驳回 → 记忆条目 importance_score 反映用户反馈（被驳回相关条目分数下降）
22. **分类置信度**：构造一条 LLM 分类不确定的复盘内容 → 写入 `pending_review` 暂存区而非直接污染记忆；用户在 UI 内确认或调整分类

### Governance
23. **LLM Key 加密**：检查数据库表，api_key 字段不可读出明文（系统凭证管理器持有）
24. **级联删除**：删除项目 → 所有需求/Thread/Message/Report/knowledge 级联清理；删除前 30s 内可撤销
25. **员工 archive 保留历史**：archive 员工后，其历史需求和 Report 仍可查；员工不再出现在派单候选

### 体验底线
26. **Onboarding 样板**：首次启动加载样板 → 用户能在不填 Key 的情况下浏览样板需求的完整思维链
27. **Replay**：在一个已完成需求上点 Replay → 用当前的 prompt + 记忆库重新跑 → 与原结果并排展示

### 量化观测指标（V1.0 自跑 20 次需求后采样）
| 指标 | 目标 | 测量方法 |
|---|---|---|
| 澄清卡片问题"用户认为有用"的比例 | ≥ 60% | 用户确认澄清时每个问题旁加 👍/👎 微评分，回收 ≥ 20 个需求样本 |
| 同类型驳回需求再次派单时，新需求引用历史教训的比例 | ≥ 70% | 派单后 LLM-as-judge（用独立 prompt）判定澄清卡片或思维链是否引用历史 lesson/pitfall；20 个对照样本 |
| 项目知识 RAG 检索 Top-K 中"被员工实际引用进思维链"的比例 | ≥ 40% | 检索时记录注入的条目 id；LLM-as-judge 判定该条目是否在交付物或思维链中显式引用；统计命中率 |
| 执行中刷新页面后，思维链历史完整恢复 | 100% | 自动化测试：随机时间点强制刷新，比对 Message 表序号是否连续无丢失 |

> 量化采样工具是 P1（见 §13）。α 阶段手动统计 20 个样本即可；β 阶段做工具化。

以上场景稳定跑通 + 量化指标达成，V1.0 即可发布。

### V1.0 验证通过 / 失败判定线

| 阶段 | 判定线 | 不达标的处置 |
|---|---|---|
| **α 阶段（4 周）** | 验收 #1–#6 全部通过 = **引擎闭环验证成功**，继续进入 β | 任一不通过：暂停 β 阶段，重新评估产品方向（澄清前置是否真有价值？记忆沉淀是否真生效？） |
| **β 阶段（+4~8 周）** | 27 条验收 + 4 个量化指标全部达成 = **V1.0 可发布** | 量化指标不达标 50% 以上：发布推迟，回到设计层（注入策略 / 记忆模型）调整；技术性验收（#7–#11 / #18–#20 / #23–#25）不通过：是工程问题，修复后重测 |

> α 阶段失败比 β 阶段失败成本低得多——这正是两阶段的核心价值。**α 不通过不要硬上 β**。

## 十三、模块优先级（α / β 两阶段排期）

> α 阶段（4 周）只做 **P0-α**——真正不可降级的核心闭环验证。
> β 阶段（+4~8 周）完成 **P0-β**（α 阶段的延展和稳定化）+ **P1**（增强能力）。
> 不做清单见 §11，本表不重复列出"V1.0 不做"项。

### P0-α — α 阶段（4 周）必做

| 模块/能力 | 降级底线 |
|---|---|
| 项目/员工/技能/需求 CRUD | 不可降级 |
| 澄清前置 + 思维链三栏 + 用户控制权 | 不可降级，验证目标本身 |
| `ask_user` 工具调用机制 | 不可降级，比文本标签可靠 10× |
| 执行中再澄清（仅"方案分歧 + 信息缺失" 2 种触发） | 不可降级，验证澄清机制 |
| 状态实时落库 + 会话恢复 | 不可降级，同步执行体验基础 |
| Budget Cap（iterations / tokens / wall_time） | 不可降级，防失控 |
| 工具错误处理（仅超时重试） | 不可降级，最小可靠性 |
| 项目介绍自动 RAG（最小可用版） | 不可降级 |
| 项目规范 conventions（required 全量注入） | 不可降级，开发型员工前提 |
| 双向沉淀 + 记忆条目自动注入 Top-K | 不可降级，差异化能力 |
| 工具注册地基（M7 仅 registry + `ask_user` 一个工具） | 不可降级，不打地基后续重构成本极高 |
| Token 累计 + 成本估算可见 | 不可降级，建立信任 |
| Thread / Message 数据模型 | 不可降级，思维链落地基础 |
| LLM Key OS 凭证加密 | 不可降级，数据安全底线 |

α 阶段单需求活跃执行（多需求排队）；不做样板项目、不做规范文件读取、不做 Importance Scoring 等。

### P0-β — β 阶段（+4~8 周）必做（α 阶段的延展与稳定化）

| 模块/能力 | 备注 |
|---|---|
| 多需求真并发（替换 α 的串行排队） | SSE 多路复用 + 全局 rate limiter |
| 复杂度阈值（simple 跳过澄清） | 让澄清不反噬体验 |
| 执行中再澄清扩展到 5 种触发 | 加入判定不准 / pitfalls 命中 / cost_alert |
| Context Length 超限自动摘要 | 长任务必发生 |
| 工具错误处理全套（死循环检测 + 格式校验 + 致命错误暂停） | 行业头号失败原因 |
| Prompt Cache 启用（稳定前缀复用） | 成本降至 ~10% |
| 项目规范 recommended 条目 RAG 检索 | α 全部按 required 处理 |
| AGENTS.md / CLAUDE.md / .cursorrules 文件读取 | α 只支持 UI 内维护 |
| Importance Scoring（简化打分公式） | α 用 LRU + hit_count |
| 分类置信度 + pending_review 暂存 | α 直接写入 + 用户手动整理 |
| Onboarding 样板项目 M8（含预录制思维链回放） | α 不做样板，要求用户自建 |
| Governance 完整（级联删撤销 + archive 保留 + 撤权立即） | α 仅做 Key 加密 |
| 自带工具扩展：web_search / read_file / write_file | α 只有 ask_user |

### P1 — β 阶段视进度做的增强

| 模块/能力 | 备注 |
|---|---|
| Replay & Debug M9（单次手动 replay） | 验证 prompt/记忆改动效果 |

| 记忆条目自动归档（低分 + 30 天未命中） | 配合 Importance Scoring |
