# AI 数字员工 — 产品需求文档

> 单文档版本。包含产品定位、版本路线、数据模型、用户流程、页面线框、各版本范围。
> 仅产品需求层面，不涉及技术设计。

---

# 第一部分：总览

## 一、产品定位与价值主张

### 1.1 一句话定位
> 让个人和小团队用「招人 + 派活」的方式管理 AI，把零散的 Agent 调用沉淀成一支可复用的数字团队。

### 1.2 与同类产品的差异
| 维度 | 主流 AI 助手 / Agent | AI 数字员工 |
|---|---|---|
| 心智模型 | 「对话」/「指令」 | 「组织 + 岗位」 |
| 复用单元 | Prompt / Agent 配置 | **员工档案**（岗位+技能+记忆+履历） |
| 工作流入口 | 用户主动提问 | **提需求 → 自动分派** |
| 多任务并行 | 难，需手动切换会话 | 多员工并行接单 |
| 沉淀资产 | 散落的会话记录 | 员工成长档案、项目工时、需求闭环 |

### 1.3 目标用户与场景
- **独立开发者**：组建「产品经理 + 前端工程师 + 设计师」三人 AI 团队，并行推进副业项目
- **创业者 / 一人公司**：用「运营专员 + 客服 + 数据分析师」处理重复性工作
- **自由职业者**：把不同领域客户的活儿分给对应「专员」，按需求工单管理交付

### 1.4 核心形态
- **真·AI 员工执行平台**：员工能调用 LLM 和工具，真实完成任务
- **核心数据关系**：员工 → 技能 → 接需求；**项目承载知识**（项目介绍自动 RAG）
- **协作形态**：支持员工经理调度多员工拆解协作（V1.1）
- **计费形态**：V1.0 用户自带 LLM Key；V1.1 引入平台代付按量结算

## 二、产品哲学

> **宁可多问，不可盲干。**
> 员工不是"接到指令就闷头跑"的工具人，而是会沟通、会确认、会暴露思考过程的协作者。
> 每一步对用户可见、可介入；犯过的错会成为下次的提醒。

体现在三个机制：
1. **澄清前置**：接单先复述理解 + 列拆解 + 提问，确认后才动手
2. **过程透明**：思维链 + 当前步骤 + 下一步计划三栏实时可见
3. **纠错沉淀**：失败 / 返工 / 差评自动复盘，写入个人教训 + 项目踩坑库

## 三、版本路线图

### V1.0 — 引擎验证版（4 周）
**定位**：纯引擎，验证核心闭环；**单用户本地使用**；用户自带 LLM Key；**不做商业化**。

包含：项目 / 员工 / 技能 / 需求 / 核心引擎（澄清+执行+思维链+总结）/ 记忆与纠错。
不包含：仪表盘、计费、用户系统、模板市场、异步执行、日报周报、员工经理、自动推荐。

### V1.1 — 协作与汇报扩展（+4 周）
- 仪表盘（首次引入）
- 账户与计费中心（平台代付落地）
- 自动技能匹配 / 员工推荐
- 员工经理 + 协作调度
- 日报 / 周报（员工 + 项目双维度）
- 工具调用扩展（联网、代码执行、文件读写）
- 异步执行 + 站内通知
- 预算控制

### V2.0 — 生态与规模化（远期）
- 模板市场（岗位模板 + 员工模板共享）
- 月报 / 季报 / 年度总结
- 外部推送（邮件、微信、飞书）
- 团队成本与 ROI 分析
- 创作者激励 / 分成
- 移动端

# 第二部分：核心概念与数据模型（全版本共享）

## 五、四大核心实体 + 一种沉淀产物

```
┌──────────┐       ┌──────────┐      ┌──────────┐
│  员工     │───┬──▶│ 技能      │      │  项目     │
│ Employee │   │   │  Skill   │      │ Project  │
└──────────┘   │   └──────────┘      └────┬─────┘
     ▲ │       │                          │
     │ │ 沉淀   │ 拥有                     │ 归属
     │ ▼       │                          ▼
     │┌──────┐ │                     ┌──────────┐
     ││汇报   │ │                     │  需求     │
     ││Report│ │                     │Requirement│
     │└──┬───┘ │                     └────┬─────┘
     │   │     │                          │
     │   └─汇总─┴──────────接单───────────┘
     └──────────────────────────────────────────
```

**核心关系**：`员工 → 技能 → 接需求`
即：员工是核心实体，员工拥有技能/岗位，员工接需求工单完成工作；项目是需求的组织容器；汇报是员工/项目沉淀产物。

## 六、记忆模型（以项目为主）

**决策**：业务知识沉淀**主归项目**，员工只保留极轻量的"自我画像 + 个人教训"。

> "员工是带着技能来项目里干活的人，知识属于项目，人属于团队。"
> 员工换项目不应该带走项目的业务知识；项目换员工不应该丢失业务积累。

### 6.1 记忆分层

| 记忆层 | 字段位置 | 角色 | 存什么 | 注入时机 |
|---|---|---|---|---|
| **项目知识 + 踩坑库** | `project.knowledge.facts` + `project.knowledge.pitfalls` | **主记忆** | 项目背景、术语、技术栈、决策、关键文档、**本项目踩过的坑** | 需求归属此项目时**自动 RAG 检索注入** |
| **员工自我画像 + 教训** | `employee.memory.style` + `employee.memory.lessons` | 辅助 | 工作风格、偏好、**个人踩坑教训** — 不存业务知识 | 该员工接任何需求都注入 |
| **需求上下文** | `requirement.description` + 附件 | 临时 | 本次任务的具体描述 | 单次执行，不沉淀 |

### 6.2 关键设计点

1. **项目介绍自动 RAG**
   - `project.description` 一经填写/编辑立即触发向量化
   - 员工接需求时按需求描述自动检索 Top-K 注入 prompt

2. **需求总结的写入流转**
   ```
   需求完成 / 被驳回 / 多次返工
              │
              ├──→ 业务事实         ──→ project.knowledge.facts
              ├──→ 项目踩坑         ──→ project.knowledge.pitfalls
              ├──→ 工作风格反思     ──→ employee.memory.style
              └──→ 个人教训         ──→ employee.memory.lessons
   ```

3. **项目归档**：`project.knowledge` 转为只读历史；新建相似项目时可一键克隆作为起点

4. **员工离职/复用**：员工记忆是"人的画像"，可跨项目复用；项目知识不带走（业务保密 + 避免串味）

### 6.3 形态层面
- **项目详情页**新增「项目知识」Tab：可浏览、编辑、手动追加项目记忆条目
- **员工详情页**的「记忆」Tab 只展示自我画像 + 个人教训
- 需求执行时，系统按 `项目知识 RAG 检索结果 + 员工自我画像 + 需求描述` 顺序拼装上下文

## 七、关键用户流程

### 流程 A：日常派需求（主流程）
```
新建需求（标题+描述+所需技能）
   ↓
（V1.0 手动指派 / V1.1 自动推荐 Top 3）
   ↓
员工生成澄清卡片
   ↓
用户确认/回答 → 员工开始执行
   ↓
用户实时观察思维链 / 中途介入 / 改下一步计划
   ↓
员工交付
   ↓
验收 ✓ → 归档 + 项目知识沉淀 + 员工记忆更新
驳回 ✗ → 带反馈打回 + 自动复盘写入教训库
```

### 流程 B：每日 / 每周看汇报（V1.1）
```
配置时间到 → 系统自动生成员工日报 + 项目日报
   ↓
仪表盘出现「X 份新日报」提醒
   ↓
用户进入汇报中心 → 浏览 / 编辑（自动已发布）
   ↓
周末汇总成周报
```

### 流程 C：培养员工
```
查看员工履历 → 发现某技能薄弱 / 教训累积多
   ↓
进入员工详情 → 添加/替换技能，或手动编辑教训库
   ↓
或编辑员工的人设 / 自我画像
```

# 第三部分：V1.0 — 引擎验证版

> **定位**：纯引擎。验证"AI 员工能不能**接活 → 澄清 → 执行 → 沉淀 → 下次更好**"这个核心闭环。
> **不是产品，是引擎**。所有商业化、运营、协作、生态能力一律推迟到 V1.1+。
> **周期目标**：4 周

## 九、V1.0 定位与边界

### 9.1 想验证什么
只验证**一个问题**：

> 让一个配置过技能和人设的 AI 员工，接到一个项目内的需求，能不能在「澄清前置 + 过程透明 + 记忆沉淀 + 纠错学习」这套机制下完成任务，并且下次同类任务做得更好？

跑通了，产品方向被验证；跑不通，就重新设计，不浪费时间在仪表盘、计费、模板市场这些"运营外壳"上。

### 9.2 单用户本地使用
- 不做用户系统 / 登录 / 多租户 / 多人协作
- 所有数据都是当前操作者的
- 用户**自带 LLM Key**（在员工配置中填入）
- 平台不收费、不限速、不计量

### 9.3 V1.0 范围

| 模块 | V1.0 | 备注 |
|---|---|---|
| 项目 CRUD + 介绍自动 RAG | ✅ | 知识容器 |
| 员工 CRUD + 配置 | ✅ | 人设、技能、模型 Key |
| 技能 CRUD + 预置 5~10 个 | ✅ | 能力单元 |
| 需求 CRUD + 简单列表 | ✅ | 任务入口 |
| **核心引擎**（澄清 / 执行 / 思维链 / 总结） | ✅ | **灵魂** |
| **记忆与纠错**（双向沉淀 + 自动注入） | ✅ | **差异化能力** |
| 仪表盘 | ❌ | V1.1 |
| 计费 / 账户 / 付费 | ❌ | V1.1 |
| 用户系统 / 登录 / 多租户 | ❌ | V2.0 之后 |
| 模板市场 | ❌ | V2.0 |
| 异步执行 / 站内通知 | ❌ | V1.1 |
| 日报 / 周报 | ❌ | V1.1 |
| 员工经理 / 协作调度 | ❌ | V1.1 |
| 自动技能匹配 / 员工推荐 | ❌ | V1.0 手动指派 |

## 十、V1.0 功能模块（六个 M）

### M1. 项目（知识容器）
- **列表**：项目名 + 简介 + 需求数
- **详情**：
  - 介绍编辑（富文本/Markdown）— 保存即触发向量化
  - 「项目知识」Tab：facts 列表 + pitfalls 列表（均可手动编辑/删除）
  - 「需求」Tab：本项目下的所有需求
- **自动 RAG**：员工接需求时按需求描述检索 Top-K 注入

### M2. 员工（能力载体）
- **列表**：头像 + 名字 + 岗位 + 技能标签 + 状态
- **详情**：
  - 基本信息（名字、岗位、人设/persona）
  - 技能挂载（多选）
  - 模型配置（provider、model、api_key、temperature、max_tokens）
  - 「记忆」Tab：style（文本可编辑）+ lessons 列表（可手动增删）
- **临时对话**：员工详情页内置对话面板，绕开需求流程直接聊（用于调试人设）
- ⚠️ V1.0 **不做**岗位模板系统，员工通过手动配置创建

### M3. 技能（能力单元）
- **列表**：name + category + description
- **编辑器**：name / category / description / prompt_template / required_tools / examples
- **系统预置（V1.0 必带）**：通用对话、文案撰写、代码生成、文档总结、联网搜索（共 5 个起步）

### M4. 需求（任务入口）
- **列表**：表格视图（标题 / 项目 / 员工 / 状态 / 优先级），按状态筛选
- **新建**：
  - 必填：标题、描述、项目、员工
  - 选填：所需技能、优先级
  - V1.0 仅支持「**同步执行**」
- **详情**：澄清卡片 → 思维链三栏 → 交付物 → 验收/驳回（详见 M5/M6）
- ⚠️ V1.0 **不做**看板视图（列表足够）

### M5. 核心引擎（V1.0 的灵魂）

#### 5.1 接需求 → 澄清卡片
员工接到需求后**不立即动手**，先生成澄清卡片（理解复述 + 拆解步骤 + 3~5 个问题）。
需求状态：`待澄清` → 用户确认/回答后 → `进行中`。

#### 5.2 执行：思维链三栏
左栏思维链流式 / 中栏当前步骤 / 右栏下一步计划（用户可编辑）。

#### 5.3 用户控制权
- ⏸ **暂停**：立即停下，可追加说明后继续
- 💬 **追问**：在思维链上插入消息
- ✏️ **改计划**：直接编辑右侧步骤列表
- ⏹ **强制结束**：终止，可选"保留产出 + 验收"或"丢弃"

#### 5.4 执行中再澄清
5 种触发场景员工**主动暂停发问**：

| 触发场景 | 员工行为 |
|---|---|
| 方案分歧 A vs B | 列选项 + 利弊 → 询问 |
| 关键信息缺失 | 说明需要什么 → 等待 |
| 判定不准 | 给 2-3 个变体 → 询问偏好 |
| 命中项目 pitfalls | "上次这里出过问题，这次想这样做" → 确认 |
| 预估成本 / 时间超阈值 | 说明预估 → 询问是否继续 |

需求状态：`进行中` → `等待回答` → 用户回复 → `进行中`。

#### 5.5 完成 → 自动总结
员工产出最终交付物，用户验收 / 驳回。完成后自动生成单需求总结，按 M6 规则分流写入记忆。

### M6. 记忆与纠错（与引擎一体）

#### 6.1 双向沉淀（每个需求总结都触发）
```
需求总结（无论验收还是驳回）
   │
   ├─ 业务事实/决策      ──→ project.knowledge.facts
   ├─ 工作风格反思        ──→ employee.memory.style
   └─ 教训/踩坑          ──→ 详见 6.2
```

#### 6.2 纠错触发条件
任一即触发自动复盘 + 写入教训库：
- 需求被**驳回**（status: 已驳回）
- 需求被**多次返工**（≥2 次修改才验收）
- 用户在交付物 / 验收时给**负面反馈**（关键词检测 + LLM 判定）

#### 6.3 教训分流
```
复盘
  ├── 我个人应该改的     → employee.memory.lessons
  │   "我容易没确认就动手，下次先问"
  │
  └── 这个项目的踩坑     → project.knowledge.pitfalls
       "客户讨厌长邮件，要求 200 字以内"
```

#### 6.4 自动注入（新需求接单时）
```
[员工人设]
[员工自我画像 style]
[相关的个人教训 lessons — 按相关性 Top-K]
[项目知识 facts — RAG 检索 Top-K]
[相关的项目踩坑 pitfalls — 按语义相似 Top-K]
[需求描述]
```

#### 6.5 用户可干预
- 员工详情页「记忆」Tab：可手动编辑 style / 增删 lessons
- 项目详情页「知识库」Tab：可手动增删 facts / pitfalls
- 用户可手动新增条目

#### 6.6 避免无限累积
- 同主题条目自动去重 + 合并
- 长期不命中的条目自动归档
- 用户可手动清除

## 十一、V1.0 明确不做（重要）

为保持引擎纯净，以下能力**明确不做**：

- ❌ 用户系统、登录、注册、找回密码
- ❌ 多用户、多团队、权限、共享
- ❌ 付费、充值、余额、账单、计量
- ❌ 仪表盘、数据看板、统计报表
- ❌ 日报 / 周报 / 月报
- ❌ 模板市场、岗位模板分享
- ❌ 员工经理、员工间转派、协作请教
- ❌ 自动技能匹配、员工推荐算法
- ❌ 异步执行、后台任务、定时任务
- ❌ 站内通知中心、邮件、短信、微信推送
- ❌ 移动端
- ❌ 数据导出（除单个需求的交付物下载）
- ❌ 国际化 / 多语言（仅中文）
- ❌ 主题切换、深色模式

如发现以上任何一项是"必须做"的，走变更评审，不能直接加。

## 十二、V1.0 验收标准（端到端 6 个场景）

1. **基础配置**：创建项目（含介绍）→ 创建员工（挂技能、填 LLM Key、写人设）→ 项目介绍向量化完成可查询
2. **澄清前置跑通**：派需求 → 员工生成澄清卡片 → 用户确认 → 员工开始执行
3. **思维链透明**：执行过程中能看到思考、工具调用、当前步骤、下一步计划；用户可暂停、追问、改计划
4. **执行中再澄清**：构造方案分歧场景 → 员工暂停发问 → 用户回答 → 员工继续
5. **记忆沉淀可见**：完成需求 → 项目知识库出现新 facts → 员工 style 有更新
6. **纠错学习闭环**：驳回需求 → 教训写入 lessons + pitfalls → 派同类型新需求 → 在澄清卡片或思维链中能看到教训被引用

这 6 个场景稳定跑通，V1.0 即可发布。

### M8. 员工经理与协作调度（V1.1 新增模块）

**8.1 员工经理（Manager）**
- 创建/编辑员工时勾选 `role_type = manager`
- 拥有 `can_delegate_to`：可调用的下属员工列表

**8.2 三种协作模式**

| 模式 | 流程 |
|---|---|
| **拆解-分派** | 经理接复合需求 → 生成澄清卡片（含拆解树）→ 用户确认 → 分派子任务给下属 → 并行/串行执行 → 经理汇总交付 |
| **同级转派** | 员工 A 执行中发起转派给 B → 经理或用户确认 → 转移所有权 |
| **协作请教** | 员工 A 执行中 @ 另一员工咨询 → 拿建议但不转移所有权 |

**8.3 可视化**
- 需求详情页展示子需求树
- 团队视图：以经理为根节点的组织树

**8.4 安全保护**
- 调用链超过 3 层或循环时自动中断
- 子任务总成本计入父需求成本上限

### M10. 账户与计费中心（V1.1 首次引入）

- **计费规则**：按 LLM token × 平台加成系数；工具调用单独计费；用户可选「自带 Key 模式」（沿用 V1.0 行为，不收费）
- **成本可见性**：仪表盘卡片 / 单需求执行后展示
- **预算控制**：单需求上限、项目级月度上限、员工级月度上限（可选）、预警阈值 50%/80%/95%
- **账单**：月度账单 PDF、消费明细可导出

## 十五、V1.1 验收标准

1. **仪表盘**：登录默认进入仪表盘，看到团队概况 + 待处理 + 余额 + 今日汇报
2. **付费**：充值 → 派需求消耗 → 余额下降 → 余额不足被阻断（自带 Key 模式不受限）
3. **经理调度**：派复合需求给经理 → 拆解为子任务 → 分派 → 并行执行 → 汇总交付
4. **转派**：员工 A 执行中转派给 B → B 接管 → 历史思维链可追溯
5. **协作请教**：员工执行中 @ 另一员工 → 拿建议 → 继续执行
6. **员工日报**：员工配置 18:00 自动生成 → 到点产出 → 仪表盘提醒
7. **项目日报**：项目自动汇总当日全员产出 → 独立于员工日报
8. **预算控制**：项目设月度上限 → 达 80% 预警 → 超出后新需求被冻结
9. **工具**：员工能主动调用联网搜索、代码执行，思维链可见
